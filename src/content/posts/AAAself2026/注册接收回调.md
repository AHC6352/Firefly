## 先理解问题

假设你在写 CAN 驱动，收到的消息存在 `msg` 里。现在要对消息做处理——比如 ID 为 `0x100` 的帧是温度数据，要打印出来。

**最直接的做法**：在驱动的收数据循环里写死处理逻辑。

```c
void CAN_Driver_Process(void)
{
    CAN_Message_t msg;
    while (CAN_Driver_GetRxMessage(&msg) == HAL_OK)
    {
        // 收到消息了，处理它
        if (msg.id == 0x100) {
            printf("温度: %.1f\r\n", msg.data[0] * 0.5f);
        }
    }
}
```

这能跑，但有问题：**下次换个项目，温度帧 ID 变了，或者不打印温度了，你就得改驱动代码。** 驱动本应是通用模块，不应该和具体业务绑定。

理想状态是：驱动只管收发，不管数据拿去干什么。那驱动怎么知道数据该交给谁处理？——**让上层告诉驱动一个"处理函数"的地址**。这就是回调。

---

## 第一步：认识函数指针

普通指针存的是变量的地址。**函数指针存的是函数的地址**——函数的入口地址，通过它可以跳转过去执行。

### 普通指针对比

```c
int a = 10;
int *p = &a;      // p 存了 a 的地址
*p = 20;          // 通过 p 找到 a，修改它
```

### 函数指针

函数名本身就是一个地址（和数组名类似），所以取函数地址时 `&` 可以省略：

```c
void MyHandler(CAN_Message_t *msg)
{
    // 处理消息...
}

// 定义一个函数指针变量，存 MyHandler 的地址
void (*func_ptr)(CAN_Message_t *) = MyHandler;

// 通过函数指针调用——效果等同于直接调用 MyHandler(&msg)
func_ptr(&msg);
```

拆解这行定义：
```
void (*func_ptr)(CAN_Message_t *)
  │    │     │         │
  │    │     │         └── 参数：一个 CAN_Message_t 指针
  │    │     └── 变量名是 func_ptr，* 表示它是指针
  │    └── 返回值是 void
```

用函数指针，你就拿到了一个"可以指向不同函数"的变量——指向谁，调用时就执行谁。

---

## 第二步：驱动怎么存这个指针

驱动需要一个变量来存函数指针。初始时什么都没存，就是 `NULL`：

```c
// 驱动文件 can_driver.c 内部

static void (*g_rx_callback)(CAN_Message_t *) = NULL;
//     ↑
//   static：这个变量只在本文件可见，外部访问不到
```

所谓"注册回调"，就是**上层调用一个函数，把自己的处理函数地址传进来，驱动把它存到 `g_rx_callback` 里**：

```c
void CAN_Driver_SetRxCallback(void (*callback)(CAN_Message_t *))
{
    g_rx_callback = callback;  // 把上层的函数地址存起来
}
```

上层调用：

```c
// main.c
CAN_Driver_SetRxCallback(MyHandler);  // 把 MyHandler 的地址传进去
```

这一步做完了，`g_rx_callback` 就指向了 `MyHandler`。

---

## 第三步：驱动在收到消息时调用它

驱动的消息处理循环里，每收到一帧就检查一下有没有注册过回调：

```c
void CAN_Driver_Process(void)
{
    CAN_Message_t msg;

    while (CAN_Driver_GetRxMessage(&msg) == HAL_OK)
    {
        if (g_rx_callback != NULL)   // 有人注册过吗？
        {
            g_rx_callback(&msg);     // 有，调用它，把消息传过去
        }
        // 没注册就跳过，消息丢弃
    }
}
```

**为什么需要 `!= NULL` 检查？**
如果上层忘记注册就启动了驱动，`g_rx_callback` 还是 `NULL`。直接调用一个地址为 0 的函数会触发 HardFault，单片机直接死机。加了 NULL 检查后，没注册时只是消息被丢弃，系统继续正常运行。

---

## 第四步：上层写处理逻辑

现在你只需要在应用代码里写一个函数，签名为 `void 函数名(CAN_Message_t *)`：

```c
void MyHandler(CAN_Message_t *msg)
{
    switch (msg->id)
    {
        case 0x100:  // 温度传感器帧
            printf("温度: %.1f°C\r\n", msg->data[0] * 0.5f);
            break;

        case 0x200:  // 电机控制帧
            Motor_Control(msg->data[1]);
            break;

        default:
            break;   // 不关心的 ID，忽略
    }
}
```

---

## 第五步：初始化时组装起来

```c
int main(void)
{
    HAL_Init();
    CAN_Driver_Init();
    CAN_Driver_SetRxCallback(MyHandler);  // 把处理函数注册进去
    CAN_Driver_Start();

    while (1)
    {
        CAN_Driver_Process();  // 有消息就调 MyHandler，没消息就空转
    }
}
```

顺序很重要：**先注册，再启动**。启动后消息就开始收了，如果回调还没注册，前几帧收不到。

---

## 完整调用链路走一遍

```
CAN 总线上来了一帧消息
         │
         ▼
CAN_Driver_Process() 被调用
         │
         ▼
GetRxMessage(&msg) —— 从硬件 FIFO 取出一帧
         │
         ▼
g_rx_callback != NULL ? —— 是，指针存了 MyHandler 的地址
         │
         ▼
g_rx_callback(&msg) —— 跳转到 MyHandler 里执行
         │
         ▼
switch(msg->id) —— 匹配到 0x100，打印温度
         │
         ▼
返回到 CAN_Driver_Process，继续取下一帧
```

## 用 typedef 让代码更可读

上面 `void (*)(CAN_Message_t *)` 这种写法看多了眼花。可以用 `typedef` 给它起个名字：

```c
// 给 "void (*)(CAN_Message_t *)" 这个类型起名叫 CAN_RxCallback
typedef void (*CAN_RxCallback)(CAN_Message_t *);
```

之后所有用到这个类型的地方都变得干净：

```c
// 之前：
static void (*g_rx_callback)(CAN_Message_t *) = NULL;
void CAN_Driver_SetRxCallback(void (*callback)(CAN_Message_t *))
void MyHandler(CAN_Message_t *msg)

// 之后：
static CAN_RxCallback g_rx_callback = NULL;   // 含义一目了然
void CAN_Driver_SetRxCallback(CAN_RxCallback callback)
// 上层定义时也可以直接用：
void MyHandler(CAN_Message_t *msg)  // 签名不变，同上
```

`typedef` 没有改变任何行为，只是让代码更好读。更重要的是，如果你不小心写错了函数签名，编译器会根据 `CAN_RxCallback` 的类型帮你检查出来。

## 对比：有回调 vs 没回调

| 场景 | 直接在驱动里写逻辑 | 用回调 |
|------|-------------------|--------|
| 改处理逻辑 | 打开 `can_driver.c` 修改 | 只改应用层的 `MyHandler` |
| 换项目复用驱动 | 删掉旧项目的逻辑，写新的 | 驱动文件直接复制，不改一行 |
| 多个模块都要同一帧 | 在收数据的地方到处加代码 | 每模块各写各的回调，各自注册 |

本质一句话：**驱动不知道也不关心上层是谁，它只管把消息递给一个函数指针。至于这个指针对应的函数做什么，是上层的事。**

## 进阶：多个模块同时收消息

单回调一次只能绑一个处理函数。如果日志模块和控制模块都想收到同一帧，怎么办？——用数组存多个指针：

```c
#define MAX_CALLBACKS 4

static CAN_RxCallback g_rx_callbacks[MAX_CALLBACKS];  // 回调指针数组
static int g_callback_count = 0;                      // 当前存了几个

// 注册：追加到数组末尾
void CAN_Driver_RegisterRxCallback(CAN_RxCallback callback)
{
    if (g_callback_count < MAX_CALLBACKS)
    {
        g_rx_callbacks[g_callback_count] = callback;
        g_callback_count++;
    }
}

// 收到消息：逐个通知
void CAN_Driver_Process(void)
{
    CAN_Message_t msg;
    while (CAN_Driver_GetRxMessage(&msg) == HAL_OK)
    {
        for (int i = 0; i < g_callback_count; i++)
        {
            g_rx_callbacks[i](&msg);  // 通知第 i 个注册者
        }
    }
}
```

这样日志和控制各注册各的，驱动收到一帧后挨个调用。注意这里**不需要 `!= NULL` 检查**——`g_callback_count` 是 0 时循环就不会进入，天然安全。
